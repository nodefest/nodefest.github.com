I tried the E2E test with ReactNative application

I think that in 2017, ReactNative (RN) is starting to build an important position in developing native applications.
Although RNs considerably absorb differences between multiple platforms such as iOS / Android, support is still weak in the UI and E2E test areas.
Unit testing can be written naturally in the Node environment if you properly isolate dependencies, but as you step into the native functions and UI, there are present conditions that are very difficult to test in the Node environment.
In a test case that can not be visualized with unit tests, it is difficult to overlook bugs in a test case, put unit tests up to the view layer, and exhausted the maintenance cost of component testing.
After all, I think that there are many teams that check the operation of the application, debugging and checking the operation with the power of the person after all.
But have you ever told us that you have confirmed human power, overlooked by accustoming or impatience, or you missed the problem you noticed if you saw it due to leakage of testing points?
How can we pass the test while limiting the verification cost in a small development team that can not secure tester and debugger resources? From the point of view, I would like to tell you the process of incorporating the test using the E2E test tool Appium in the practical RN project, the know-how as a result of the test.

A Toy Robot and a Functional Pattern

A Functional Programming-inspired pattern to separate instructions and interpretations.
Inspiration gained from functional programming separates instruction and interpretation.

We can learn a lot of useful code patterns from Functional Programming. Some of our code into separate 'instructions' and 'interpreters'. By keeping them distinct, we can more easily re-use our code, and make it easier to test. This talk will show you how to use this technique: we'll start with simple examples, dig into real-world libraries like 'reducex-saga' and 'fluture', and finally drive the wheels of a toy robot .

Native ES Module - something almost, but not quite entirely unlike CommonJS

CommonJS, along with the npm registry, has become the de facto standard in Frontend and NodeJS development today. But Native ES Modules are coming to NodeJS and the browser. On the surface, they look incredibly similar to CommonJS modules, but the differences are important.In this talk I will describe these differences, the consequences of these differences, and how to start preparing for the new world of Native ES Modules.
